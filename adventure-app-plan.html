비전 & 컨셉

왜 이 앱을 만드는가, 누구를 위한 앱인가

앱의 정체성

이 앱은 단순한 여행 가이드가 아니다. 모험가를 위한 도구다. 구글맵에 나와있는 식당 리스트나 관광 명소를 보여주는 것이 목적이 아니라, 여행을 오래 다녀온 사람도 몰랐을 숨겨진 장소를 발굴하고, 이동 수단의 물리적 한계를 고려한 현실적인 일정을 만들어주며, 그 장소에 다녀간 사람들이 남긴 진짜 이야기를 들려주는 플랫폼이다.

유럽에는 도시국가 수준의 작은 마을, 성벽 위의 카페, 폭포 뒤의 동굴이 있다. 우리나라 사람은 물론 현지인도 잘 모르는 그런 곳. 그 장소를 AI가 추천하고, 거기 다녀온 누군가가 남긴 방명록이 기다리고 있다면 어떨까. 그것이 이 앱의 핵심 경험이다.

03

지도 시스템

구글맵 없이 구현하는 자체 지도 — OSM + MapLibre GL

왜 구글맵을 쓰지 않는가

💸 비용 문제
구글맵 API는 월 200달러 무료 크레딧이 있지만, 사용자가 늘면 기하급수적으로 증가한다. 지도 렌더링(Maps JS API)만으로도 월 1000명 DAU 기준 약 50만~200만 원 이상 발생 가능. POI 상세정보(Places API)까지 쓰면 단가는 더 올라간다. 이 앱은 구글맵 없이도 동등한 수준의 지도를 자체 구현한다.

지도 기술 스택

OSM (OpenStreetMap)
세계 최대 오픈소스 지도 데이터. 위키피디아처럼 전 세계 수백만 명이 편집하는 지도. 상업적 사용 무료 (ODbL 라이선스). 구글보다 일부 지역은 오히려 더 정밀하다.
MapLibre GL JS
Mapbox GL JS의 완전 오픈소스 포크. 벡터 타일 기반의 GPU 가속 렌더링. 미니멀하고 커스텀한 지도 스타일 완전 제어 가능. 상업적 사용 무료.
타일 서버 옵션
개발 단계: Stadia Maps, Maptiler (무료 플랜)
성장 단계: 자체 호스팅 (tileserver-gl + planetpbf 파일)
자체 서버 구축 시 월 비용은 서버비 고정, 요청 수 무관.

지도 레이어 구성

사용자 레이어 (User Layer)
방명록 마커 내 위치 현재 경로 저장한 장소
↓
POI 레이어 (Data Layer)
일반 POI 핀 숨겨진 명소 💎 클러스터링 카테고리 필터
↓
지도 기반 레이어 (Map Layer)
벡터 타일 (OSM) 지형 데이터 도로/건물/자연물

미니멀 지도 스타일 설계 방향

  • 배경: 짙은 다크톤 또는 웜 베이지. 지도가 배경이 되어 POI가 돋보이게.
  • 글씨: 최소한의 도시명, 도로명만. 불필요한 레이블 제거.
  • POI 핀: 카테고리별 컬러 코딩. 숨겨진 명소는 💎 특별 아이콘.
  • 경로: 이동 수단별 다른 색상. 도보(주황), 자전거(초록), 차(파랑).
  • 클러스터링: 줌아웃 시 POI를 그룹화해서 퍼포먼스 유지.

POI란 무엇인가

POI (Point of Interest) — 관심 지점. 지도 위의 모든 장소다. 식당, 카페, 박물관, 폭포, 동굴, 전망대, 성벽, 공원, 숨겨진 해변… 구글맵에는 약 30억 개 이상의 POI가 있다. 이 앱은 오픈소스 및 공개 API를 통해 약 15억 개 수준의 POI를 자체 수집·보유하는 것을 목표로 한다.

중복 제거 전략

여러 소스에서 같은 장소가 다른 이름/좌표로 올 수 있다. 예시: 경복궁 = Gyeongbokgung = Gyeongbok Palace.

python — 중복 탐지 로직 (개념)
def is_duplicate(poi_a, poi_b) -> bool: # 1단계: 좌표 거리 체크 (50m 이내) distance = haversine(poi_a.lat, poi_a.lng, poi_b.lat, poi_b.lng) if distance > 50: return False # 2단계: 이름 유사도 체크 (fuzzy match) name_score = fuzz.ratio( normalize(poi_a.name), normalize(poi_b.name) ) if name_score < 70: return False # 3단계: 카테고리 호환성 체크 return categories_compatible(poi_a.category, poi_b.category) # PostGIS로 공간 인덱스 활용해 50m 반경 후보 빠르게 조회 SELECT * FROM pois WHERE ST_DWithin( geom, ST_MakePoint(126.977, 37.579)::geography, 50 -- 50 미터 );

AI 여행일정 추천

사용자의 조건을 분석해 현실적이고 다양한 일정을 자동 생성

입력 파라미터 (사용자가 설정하는 것들)

📅
여행 기간
1일~30일. 기간에 따라 가능한 이동 범위와 일정 밀도가 자동으로 계산된다.
🚗
이동 수단
도보 / 자전거 / 오토바이 / 자동차. 각각 하루 이동 가능 거리와 허용 구간이 다르다.
🎯
여행 스타일
자연/도시/역사/음식/모험/힐링. 스타일에 따라 추천 POI 종류가 달라진다.
💰
예산 수준
저예산 / 중간 / 여유. 숙소, 식사, 입장료 수준의 POI를 조정한다.
💪
체력 수준
하루 최대 도보 거리, 언덕/계단 허용 여부. 고령자나 장애인도 배려 가능.
🌍
목적지 범위
도시 중심 / 반경 50km / 국가 전체. 탐험 범위를 설정한다.

이동 수단별 물리적 한계

이동 수단 하루 이동 거리 POI 간 최대 거리 특이사항
🚶 도보 10~15km 2~3km 경사도, 날씨, 체력 고려. 도심 집중 일정.
🚲 자전거 30~60km 10~15km 자전거 도로 유무 확인. 언덕 구간 가중치.
🏍️ 오토바이 150~300km 50km 고속도로 제한 여부, 주차 가능 여부 고려.
🚗 자동차 200~500km 100km 주차 비용/가용성, 고속도로 시간 포함.
💡 핵심 포인트
단순히 "30분 거리"로 계산하지 않는다. 도보라면 실제 보행 경로의 고도차, 자전거라면 자전거 도로 유무, 자동차라면 주차 가능 여부까지 실제 라우팅 데이터로 계산해 현실적인 일정을 만든다.

일정 생성 프로세스

Step 1
후보 POI 풀 구성
목적지 + 이동 수단 + 여행 스타일을 기반으로 PostGIS 공간 쿼리로 후보 POI를 수백 개 추출. 품질 스코어, 숨겨진 명소 여부, 사용자 방명록 활동량도 고려.
PostGIS ST_DWithin 벡터 유사도 검색
Step 2
물리적 한계 필터링
이동 수단별 하루 이동 한계를 적용. 도보 여행인데 하루에 50km 이동하는 일정 자동 제거. 각 POI 사이의 실제 이동 시간을 라우팅 엔진으로 계산.
OSRM Valhalla 거리 행렬
Step 3
TSP 동선 최적화
선택된 POI 목록을 최단 경로로 정렬. NP-hard 문제이므로 근사 알고리즘(Nearest Neighbor + 2-opt) 사용. 하루별로 나눠 지역 최적화.
TSP 2-opt OR-Tools
Step 4
AI 일정 다양화 (LLM)
GPT-4 또는 Claude에게 일정 초안을 주고 "숨겨진 명소 포함 3가지 변형 일정"을 생성하도록 요청. 유명 관광지 위주 vs 숨겨진 명소 위주 vs 혼합 버전.
LLM Chain-of-Thought RAG
Step 5
일정 카드 생성 & 제공
각 일정을 시각적 카드로 표현. 지도 위에 경로 표시. PDF/공유 링크 export. 사용자가 POI 추가/제거하면 실시간 재계산.
React MapLibre PDF export
06

동선 최적화 (라우팅)

TSP + 오픈소스 라우팅 엔진으로 현실적인 경로 계산

라우팅 엔진 선택

추천
OSRM (Open Source Routing Machine)
가장 빠른 오픈소스 라우팅 엔진. OSM 데이터 기반. 자동차/자전거/도보 프로파일 기본 제공. 응답 속도 수 ms 수준. 자체 서버 운영 가능.
고급
Valhalla
Mapbox가 오픈소스로 공개한 라우팅 엔진. 등고선/지형 고려 가능. 자전거 경로의 경사도 계산, 오토바이 경로에 강함. Matrix API 지원으로 여러 POI 간 거리 행렬 계산 용이.
대안
GraphHopper
Java 기반 라우팅 엔진. 사용자 정의 프로파일 작성 가능. 전기차, 승마, 스키 등 특수 이동 수단도 지원 가능. 커뮤니티 활발.

TSP (외판원 문제) 쉽게 이해하기

TSP란?
"서울, 부산, 제주, 경주를 모두 들르는데 총 이동 거리를 최소화하려면 어떤 순서로 가야 하나?" 이 문제다. POI가 10개면 경우의 수가 약 360만 가지. 100개면 계산 불가능한 수준. 그래서 근사 알고리즘을 쓴다: 정확한 최적이 아니라 90~95% 수준의 좋은 해를 빠르게 구한다.
  • Nearest Neighbor: 현재 위치에서 가장 가까운 미방문 POI로 이동. 빠르지만 최적은 아님. 첫 번째 근사해 생성에 사용.
  • 2-opt 개선: 경로에서 두 구간을 바꿔보면서 더 짧아지면 적용. 수십 번 반복해 품질 향상.
  • Google OR-Tools: 구글이 오픈소스로 공개한 최적화 라이브러리. 제약 조건(시간, 거리, 영업시간) 포함한 VRP(Vehicle Routing Problem) 해결 가능.
  • 하루 단위 분리: 전체 일정을 하루별 서브 문제로 나눠 각각 최적화. 현실적 하루 일정 보장.

영업시간 & 현실 조건 반영

  • 영업시간: OSM의 opening_hours 태그 파싱. 아침 9시에 문 여는 박물관을 오전 1번째 일정으로 배치.
  • 이동 시간: 실제 라우팅 엔진으로 계산한 분 단위 이동 시간 포함. 직선거리가 아니라 실제 도로/경로 기준.
  • 체류 시간: POI 카테고리별 평균 체류 시간 (박물관 90분, 카페 30분, 전망대 20분 등) 자동 배정.
  • 식사 시간: 점심 12~13시, 저녁 18~19시 자동 배치. 주변 식당 POI에서 선택.

위치 기반 방명록

그 장소에 실제로 가야만 읽고 쓸 수 있는 진짜 이야기

핵심 컨셉

일반 여행 앱의 리뷰는 누구나 쓸 수 있다. 가본 적 없어도, 심지어 가짜로도. 이 방명록은 다르다. GPS로 해당 장소 20m 이내에 있을 때만 읽고 쓸 수 있다. 실제로 발을 딛고 그 공기를 마신 사람만 남길 수 있는 진짜 기록.

✨ 왜 20m인가?
GPS 정확도가 스마트폰 기준 5~15m 수준이다. 20m로 설정하면 실내에서도 인식 가능하고, 오차 범위를 감안해도 "그 장소 안에 있음"을 충분히 보장한다. 50m는 너무 넓고 (골목 하나 건너도 인식), 10m는 너무 좁아 실내에서 GPS 오차로 실패할 수 있다.

방명록 기능 상세

📝 작성
텍스트 + 사진 첨부 가능. 방문 날짜 자동 기록. 익명 또는 닉네임 선택 가능. 최대 500자. 스티커/이모지 지원으로 간단한 감정 표현 가능.
👁️ 열람 조건
해당 장소 20m 이내에 있을 때만 읽기 가능. 위치 확인은 서버에서 검증. "이 장소의 방명록 N개" 미리보기는 앱에서 보이지만 내용은 가서야 열람.
🔐 위치 검증
클라이언트 GPS만 믿으면 조작 가능. 서버에서 제출 시점의 IP 기반 대략 위치와 GPS 좌표를 교차 검증. 이상 탐지 시 경고 후 차단.
❤️ 소셜 요소
방명록에 공감(♡) 가능. 공감은 위치 제한 없이 가능. 방명록이 많은 POI는 "사람들이 많이 찾는 숨겨진 명소"로 자동 승급.

데이터 모델

sql — 방명록 테이블 구조
CREATE TABLE guestbook_entries ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), poi_id UUID REFERENCES pois(id), user_id UUID REFERENCES users(id), content TEXT NOT NULL CHECK(length(content) <= 500), photos TEXT[], -- S3 URL 배열 verified_lat DECIMAL(9,6), -- 서버 검증된 위도 verified_lng DECIMAL(9,6), -- 서버 검증된 경도 is_anonymous BOOLEAN DEFAULT false, likes_count INTEGER DEFAULT 0, created_at TIMESTAMPTZ DEFAULT now(), visit_date DATE -- 방문 날짜 (자동 기록) ); -- 공간 인덱스로 근처 POI 빠른 조회 CREATE INDEX idx_pois_geom ON pois USING GIST(geom);

숨겨진 명소 발굴 시스템

여행 고수도 모르는 장소를 AI와 데이터로 찾아내는 방법

숨겨진 명소의 정의

단순히 방문자 수가 적은 곳이 아니다. 실제로 가볼 만한 가치가 있지만 알려지지 않은 장소다. 버려진 공장, 접근하기 어려운 폭포, 지역 주민만 아는 전망대, 소셜미디어에 몇 장 없는 성벽 위 카페. 이런 곳을 데이터적으로 발굴하는 것이 이 기능의 핵심이다.

Hidden Gem 스코어링 알고리즘

지표가중치측정 방법
OSM 편집 이력 +20점 OSM에 등록돼 있지만 버전 수가 적음 → 아직 덜 알려짐
소셜미디어 언급 수 역비례 인스타그램 해시태그 수 적을수록 점수 높음
구글 리뷰 수 역비례 리뷰 수 100 미만이지만 평점 4.0 이상이면 보너스
Wikidata 여부 +15점 위키데이터에 등록된 역사적/문화적 장소 우대
접근 난이도 +10점 대중교통 미연결, 비포장도로, 도보만 가능한 경우
방명록 활동 +25점 앱 내 방명록 수 많지만 외부 인지도 낮으면 신호

예시: 유럽 숨겨진 명소 유형

  • 도시국가: 산마리노, 리히텐슈타인, 모나코 내 관광객 모르는 골목과 요새. 유럽 사람도 "거기까지 갔어?"라고 놀라는 곳.
  • 체코 보헤미아 지방: 프라하 대신 체스키 크룸로프 주변 마을. 구글 리뷰 20개 미만의 성터.
  • 슬로베니아 소코차 계곡: 에메랄드 강과 로마 다리. 여행 유튜버도 지나치는 마을들.
  • 스페인 내륙: 세고비아나 살라망카보다 더 작은 중세 마을. 성벽으로 둘러싸인 완전히 보존된 구시가.
  • 한국 내 숨겨진 명소: OSM에만 있는 지역 산책로, 공공데이터의 전통 샘물, 지역 주민 방명록이 있는 무명 전망대.

POI 품질 관리

  • 폐업 감지: Foursquare의 closed 플래그, 사용자 신고, OSM 변경 감지로 폐업 POI 자동 숨김.

  • 사용자 신고 시스템: 앱 내 "잘못된 정보 신고" 기능. 신고 누적 시 자동 검토 대기열 진입.

핵심 화면 흐름

  • 홈 (지도): 현재 위치 주변 POI + 숨겨진 명소 표시. 하단 검색바.
  • 탐험 (Explore): 카테고리/거리/이동수단 필터로 POI 목록. 지도/리스트 뷰 전환.
  • 일정 만들기: 파라미터 입력 → AI 3가지 일정 옵션 → 선택 → 편집 → 저장/공유.
  • 방명록: 현재 위치 근처 POI 목록 → 방명록 읽기 (20m 내) → 쓰기.
  • 내 여행: 저장한 장소, 만든 일정, 내가 쓴 방명록 모아보기.
